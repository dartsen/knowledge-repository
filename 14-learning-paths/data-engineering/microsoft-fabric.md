# Microsoft Fabric Learning Path

| Stage | Focus | Practice artifact |
|---|---|---|
| 1. Platform | Fabric experiences, OneLake, workspace, item, capacity, tenant | Labeled platform map |
| 2. Data product | Grain, keys, source, quality, freshness, ownership, consumers | Reviewed data contract |
| 3. Architecture | Lakehouse/warehouse/RTI choice, medallion layers, shortcut/copy, semantic path | Architecture decision and flow |
| 4. Implementation | Ingestion, incremental processing, tests, semantic model, Git/deployment | Nonproduction product with evidence |
| 5. Operations | Runs, quality, refresh, capacity, throttling, replay, recovery | Dashboard, runbook, recovery exercise |
| 6. Governance | Tenant/workspace/item/data controls, lineage, lifecycle, cost | Production-readiness review |
| 7. Leadership | Domain portfolio, capacity economics, standards, adoption, platform roadmap | Prioritized operating roadmap |

Read the [canonical family](../../03-data-engineering/microsoft-fabric/README.md) in order. Build the [sales lakehouse example](../../15-examples/data-engineering/fabric-sales-lakehouse.md), inject late and duplicate source data, deploy through separate stages, and demonstrate both data reconciliation and capacity diagnosis.

Completion means you can select Fabric workloads from requirements, design secure data-product boundaries, implement trustworthy incremental delivery, operate shared capacity, and qualify current product support instead of assuming feature parity.
